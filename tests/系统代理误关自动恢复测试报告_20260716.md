## 一、问题复现证据

2026-07-16 11:21:42，断网急救记录到以下同一轮状态变化：

- Windows 系统代理由开启变为关闭；
- 关闭前系统代理 Owner 为龙猫云 Lite，地址为 `127.0.0.1:7890`；
- Clash Party 从核心列表消失；
- 龙猫云 Lite 的 `lmclientCore` 仍以 PID 17828 监听 7890；
- 龙猫云代理核心仍可联网，但 Chrome 等依赖 Windows 系统代理的程序转为直连；
- Codex 因进程环境变量仍指向 `127.0.0.1:7890`，所以继续可以联网。

结论：Clash Party 退出清理时关闭了共享的 Windows 系统代理开关，龙猫云核心没有退出，但失去了对依赖 WinINET 的应用的接管。

## 二、修复内容

### 2.1 自动恢复

断网急救新增“系统代理被其他客户端退出误关”的自动恢复分支。只有同时满足以下条件才会重开：

1. 系统代理经过防抖确认后由开启变为关闭；
2. 关闭前 Owner、代理地址、监听 PID 和进程归属明确；
3. 关闭前代理联网正常；
4. 退出的是另一个非主客户端；
5. 原 Owner 仍由同一 PID 和进程监听原端口；
6. 系统代理地址没有变化；
7. 没有 TUN 和代理守护争抢；
8. 后台再次通过原端口完成代理联网检测。

满足条件后只恢复当前用户的 `ProxyEnable=1`，并刷新 WinINET；不会结束进程、停止服务、修改 TUN、DNS 或节点。

### 2.2 禁止自动恢复的边界

以下情况保持只读并提醒，不自动重开：

- 用户单独手动关闭系统代理；
- 主代理核心自身退出；
- 7890 的监听 PID 或进程发生变化；
- 原代理联网复核失败；
- 已出现 TUN 接管或代理守护。

### 2.3 诊断纠正

Windows 系统代理关闭、没有 TUN、但 Codex 或终端仍通过环境变量走本地代理时，诊断由错误的 `HealthySingleOwner` 改为 `ApplicationPathSplit`。状态摘要显示“Windows 未由代理接管；仅部分应用使用龙猫云 Lite”，不再把核心存活误写成全局代理接管正常。
