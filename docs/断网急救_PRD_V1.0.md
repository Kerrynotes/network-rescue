# 断网急救产品需求文档

> 文档版本：V1.0  
> 应用版本：v0.4.0-beta  
> 更新日期：2026-07-15  
> 适用平台：Windows 10 / Windows 11

## 一、产品定义

断网急救是一款面向安装了一个或多个代理客户端的 Windows 用户的网络清场工具。

它解决的不是节点选择或机场线路质量问题，而是客户端退出不完整、同源运行时互相干扰、系统代理和 TUN 残留、旧核心占用端口等本机接管冲突。

v0.4.0-beta 只保留一个面向新手的核心动作：

> 一键退出全部代理并恢复普通网络。

执行成功后只提示：

> 普通网络已经恢复；如需代理，请只打开一个客户端。

工具不选择要保留的客户端，不替用户启动代理客户端，不登录账号，不选择节点，也不重新开启代理。

## 二、问题背景

目标用户通常同时安装 Clash Verge、FlClash，以及基于 FlClash 二次开发的机场客户端。常见故障包括：

1. 代理界面退出后，核心进程仍监听 7890，另一个客户端无法接管；
2. 多个客户端分别接管系统代理、TUN、DNS、WinHTTP 或环境变量，导致出口混乱；
3. 多个 FlClash 同源客户端共享 Helper 端口或相似服务生命周期，一个客户端退出时影响另一个客户端；
4. 客户端界面和核心仍在，但动态本地 IPC 端口已经无人监听；
5. Windows 系统代理仍指向无人监听的回环端口，浏览器无法联网；
6. Codex、终端等进程仍读取旧的 `HTTP_PROXY`、`HTTPS_PROXY` 或 `ALL_PROXY`；
7. TUN、`198.18.*` 路由或代理 DNS 在客户端异常退出后残留。
8. Clash Party 等客户端使用通用 `mihomo.exe`，如果只按进程名判断，可能被误归属为 Clash Verge。

同源客户端冲突不能仅靠把 7890 改成 7892 彻底解决。可靠切换方式是：

> 全部清场 → 确认普通网络恢复 → 只打开一个客户端。

## 三、目标与边界

### 3.1 产品目标

- 用一个动作退出全部已知代理组件并恢复普通网络；
- 准确识别同源运行时冲突、客户端 IPC 失效和孤儿核心；
- 只结束白名单内、重新核对身份后的已知进程；
- 对系统代理、代理残留和直连健康分别验收；
- 权限不足、UAC 取消或残留未清理时明确显示“部分完成”；
- 保留低风险无人值守急救，不在后台自动结束进程或清理 TUN。

### 3.2 不做的事情

- 不评价或推荐具体机场服务；
- 不自动登录、选择节点、更新订阅或重连；
- 不自动启动目标客户端；
- 不调用机场客户端自己的 `/stop` 或共享 Helper 接口；
- 不结束未知进程；
- 不删除未知网卡、未知路由或企业 PAC；
- 不把单个网站失败判定为断网；
- 不把同一客户端同时开启系统代理和 TUN 直接判定为多客户端冲突。

## 四、核心用户流程

### 4.1 托盘菜单

一级菜单固定为四项：

1. 网络状态；
2. 一键退出全部代理并恢复普通网络；
3. 系统代理异常时自动急救；
4. 退出断网急救。

Codex 代理同步、诊断报告、运行记录、DNS 重置、停止单个服务等能力仅保留内部排查或 CLI 兼容入口，不占用新手托盘菜单。

“网络状态”不能只展示内部诊断字段。检测到风险时必须列出具体冲突，并提供“退出全部代理并恢复普通网络”和“暂不处理”；选择修复后仍进入一键清场的对象确认步骤。

基础网络状态必须使用“项目 / 当前状态”两列表格展示，至少包含主要代理、诊断结果、Windows 代理开关、Owner、代理地址、TUN、代理联网、绕过系统代理和 DNS。诊断说明、风险项和建议必须与基础状态分区，风险项按编号逐条显示，不能把所有字段拼成连续段落。

### 4.2 一键清场 `EmergencyDirect`

1. 保存系统代理、PAC、WinHTTP、代理环境变量、进程、服务、端口、TUN、路由和 DNS 快照；
2. 列出准备关闭的客户端、核心、Helper、服务、端口和 TUN，并进行一次产品内确认；
3. 立即关闭当前用户的 Windows 系统代理；
4. 向全部已知代理 UI 发送正常关闭请求，统一等待最多 5 秒；
5. 停止已知代理服务；
6. 对残留进程重新核对 PID、进程名、路径、启动时间和适配器白名单后精确结束；
7. 定向清理已知客户端 TUN、`198.18.*` 路由和代理 DNS；
8. 清理指向本地回环端口的 WinHTTP 及用户级、机器级代理环境变量；
9. 刷新 DNS 缓存；
10. 验证已知 UI、核心、Helper 服务、代理端口、TUN 和本地代理配置均已释放；
11. 使用 Windows 联网探针和国内探针确认普通网络；
12. 分别显示“系统代理已关闭”“代理残留已清理”“普通网络已确认恢复”。

普通权限先执行可完成步骤。只有发现受保护核心、服务、TUN、机器级环境变量或 WinHTTP 残留时，才调用断网急救 Helper 或请求一次 UAC。

### 4.3 低风险自动急救

#### 4.3.1 关闭无人监听的失效系统代理 `RestoreDirect`

后台自动关闭必须同时满足：

1. Windows 系统代理已开启；
2. 地址是本地回环代理；
3. 端口连续 3 次无人监听；
4. 普通直连确认正常；
5. 没有检测到客户端代理守护正在抢写设置。

#### 4.3.2 恢复被其他客户端退出误关的系统代理

后台自动重开必须同时满足：

1. 断网急救已经连续确认 Windows 系统代理从开启变为关闭；
2. 关闭前的系统代理 Owner、回环端口、监听 PID 和进程归属明确；
3. 关闭前该系统代理已经通过代理联网检测；
4. 同一状态变化中，退出的是另一个非主客户端，原 Owner 的核心仍在运行；
5. 原 Owner 仍由同一 PID、同一进程监听原端口；
6. 系统代理地址没有被改写；
7. 没有 TUN 接管，也没有代理守护争抢；
8. 后台通过原端口再次完成代理联网检测。

满足全部条件后，只把当前用户的 `ProxyEnable` 恢复为 `1` 并刷新 WinINET。以下情况不得自动重开：用户单独手动关闭系统代理、主代理本身退出、端口监听者变化、代理联网失败、出现 TUN 或代理守护。

这两类自动急救均不结束进程、不停止服务、不清理 TUN、不修改 DNS、不切换节点。

自动模式只关闭当前用户的 `ProxyEnable` 并广播设置变化。它不得结束进程、停止服务、清理 TUN、修改 DNS、清除环境变量或处理线路超时。

## 五、兼容模式

| 模式 | v1.0 行为 |
|---|---|
| `EmergencyDirect` | 公开的一键清场模式 |
| `StopAllClients` | 兼容映射到 `EmergencyDirect` |
| `RestoreDirect` | 保留低风险兼容能力 |
| `PrepareSwitch` | 停止执行，提示全部清场后手动只打开一个客户端 |
| `StopOtherClients` | 停止执行，提示全部清场后手动只打开一个客户端 |
| `SyncApplicationProxy` | CLI 兼容入口 |
| `ClearApplicationProxy` | CLI 兼容入口 |
| `ResetDns` | CLI 高级维护入口 |

## 六、诊断模型

扫描器统一输出 `NetworkSnapshot`、`PathHealth` 和 `Diagnosis`，并保持 v0.3 字段兼容。

### 6.1 关键状态

| 状态 | 含义 | 建议 |
|---|---|---|
| `HealthyDirect` | 普通网络正常，没有有效代理 Owner | 不操作 |
| `HealthySingleOwner` | 一个客户端正常接管 | 不操作 |
| `MixedModeSameOwner` | 同一客户端同时使用系统代理和 TUN | 仅说明 |
| `StaleSystemProxy` | 系统代理指向无人监听端口 | 满足连续确认条件后只关闭系统代理 |
| `SharedRuntimeConflict` | 两个以上 FlClash 同源客户端同时存在 UI 或 Core | 一键清场后只打开一个 |
| `ClientIpcBroken` | UI/Core 存在，但核心命令行指定的动态控制端口无人监听 | 一键清场后重新打开一个 |
| `OrphanCore` | UI 已退出，核心仍监听 7890 等代理端口 | 确认后一键清场 |
| `MultiOwnerConflict` | 系统代理、TUN、DNS、WinHTTP 或环境变量属于不同客户端 | 一键清场 |
| `TunResidual` | 核心已退出，已知 TUN 仍接管 | 确认后高权限清理 |
| `ProxyGuardConflict` | 客户端代理守护反复写回设置 | 提示关闭代理守护或完整退出 |
| `ApplicationPathSplit` | 浏览器和 Codex/终端走不同代理路径 | CLI 同步或清除旧变量 |
| `ProxyPathDegraded` | 本地端口正常、直连正常，但多个代理目标失败 | 只记录并提示更新节点 |
| `LocalNetworkFailure` | 代理和直连都失败 | 不反复修改代理，检查物理网络 |
| `ProxyResidue` | 清场后仍有已知服务、核心、TUN 或本地代理配置 | 显示部分完成 |
| `Unknown` | 证据不足 | 只生成报告，不修改未知对象 |

### 6.2 适配器字段

每个客户端适配器必须声明：

- `runtimeFamily`；
- `sharedControlPorts`；
- `controlPortFromCoreArgs`；
- UI、Core、Helper 进程规则；
- 可选的 UI、Core、Helper 可执行文件路径规则；通用 `mihomo.exe` 等共享核心名称必须声明路径规则；
- 服务名称规则；
- 默认代理端口和协议能力；
- TUN 网卡、路由和 DNS 特征；
- 代理守护识别规则；
- 可安全停止的进程白名单。
- 通用进程名的安全停止路径白名单。

FlClash 同源客户端使用 `runtimeFamily=flclash`，共享控制端口记录 47890，并从核心命令行末尾识别动态 IPC 端口。

新增客户端优先只增加适配器和脱敏测试样本，不修改主扫描逻辑。

## 七、安全与权限

- 普通监控、系统代理自动急救和用户级设置不需要管理员权限；
- Helper 安装时出现一次 UAC；
- 安装后高权限操作不再弹 UAC，但每次清场仍显示产品内确认；
- 没有 Helper 时，仅在普通权限清理后仍发现受保护残留才按次请求 UAC；
- UAC 取消、Helper 不可用或任何最终验收项失败时，不得显示恢复完成；
- Helper 只接受 `ClientId` 和固定动作，不接受任意进程名、服务名、路径或命令；
- 进程结束前重新核对 PID、名称、路径、启动时间和适配器白名单；
- 日志不记录账号、订阅、节点名称、公网 IP、Cookie 或令牌。
- 普通日志单文件最大 2 MB，结构化事件单文件最大 5 MB，均只保留 3 份归档；状态快照使用覆盖写入。
- 龙猫云断连记录只保留最近 180 天且最多 5000 条，避免长期后台运行无限增长。

## 九、验收标准

### 9.1 自动测试

- 两个 FlClash 同源客户端同时运行时识别 `SharedRuntimeConflict`；
- 回放 7890 消失、IPC 12460 失效、Helper 停止、重启后 IPC 13145 恢复的脱敏样本；
- Clash Verge UI 退出但 `verge-mihomo` 监听 7890 时识别 `OrphanCore`；
- Clash Party 的 `resources\sidecar\mihomo.exe`正确归属 Clash Party，且不能被 Clash Verge 适配器抢先匹配；
- 其他路径或缺少路径证据的 `mihomo.exe`保持未知并拒绝结束；
- 未知进程占用 7890 时拒绝结束；
- 仅 Helper 服务存在时不判为网络 Owner，但清场验收不得忽略；
- 同一客户端同时开启系统代理和 TUN 不判为多客户端冲突；
- Windows 系统代理关闭、没有 TUN、但 Codex 或终端仍通过环境变量走本地代理时，必须判定为 `ApplicationPathSplit`，不得显示 `HealthySingleOwner`；
- 非主客户端退出并误关主代理时，只有原 Owner、端口、PID 和代理联网全部复核通过才自动恢复系统代理；
- 用户手动关闭系统代理、主代理退出或端口 Owner 变化时不得自动重开；
- UAC 取消、Helper 未安装和部分清理失败时不显示完成；
- PowerShell 5.1 解析、自检、安装升级、卸载、包内校验和无启动沙箱安装通过。

### 9.2 本机实测

- 执行一次真实 `EmergencyDirect`；
- 全部已知代理 UI、核心和 Helper 服务退出；
- `ProxyEnable=0`；
- 7890/7892 没有代理监听；
- 没有已知 TUN 默认路由；
- 普通网络探针成功；
- 旧环境变量被清理，并提示重启 Codex/终端；
- 用户随后手动打开一个客户端时，该客户端能够正常接管。

### 9.3 发布门槛

- PRD、README、使用说明、更新日志和测试报告同步；
- 生成 v0.4.0-beta 中文分享包和 SHA-256；
- 保留 v0.3.3-beta 回退包；
- 安装升级不覆盖现有日志和备份。

## 十、当前实现结论

v0.4.0-beta 已将产品从“选择目标客户端进行安全切换”收敛为“全部清场后手动只打开一个客户端”。这是对多客户端共享运行时、核心生命周期分离和高权限服务残留的共同处理方式，也是新手最容易理解和验证的恢复路径。
