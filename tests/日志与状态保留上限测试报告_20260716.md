# 断网急救日志与状态保留上限测试报告

## 测试目标

确认托盘诊断入口删除后，后台排查数据仍然可用，但不会因长期运行无限占用磁盘。

## 保留策略

| 数据 | 上限 | 保留方式 |
|---|---:|---|
| `monitor.log` | 2 MB | 当前文件加 3 份归档 |
| `monitor-events.jsonl` | 5 MB | 当前文件加 3 份归档 |
| `repair-actions.log` | 2 MB | 当前文件加 3 份归档 |
| `龙猫云断连监控.log` | 2 MB | 当前文件加 3 份归档 |
| `helper.log` | 2 MB | 当前文件加 3 份归档 |
| `龙猫云断连记录.csv` | 180 天且最多 5000 条 | 每次新增断连事件后清理超期和超量记录 |
| 当前状态与最后修复结果 | 单份 | 每次覆盖写入，不累积历史快照 |

## 验证结果

- Windows PowerShell 5.1 对 `Watch-NetworkOwnership.ps1`、`Monitor-LongmaoConnection.ps1` 和 `Repair-Network.ps1` 语法解析通过。
- 托盘监控自检通过，覆盖文件锁重试和最多 3 份归档的轮转边界。
- 龙猫云监控自检通过，覆盖日志轮转、180 天时间边界和最多 5000 条的条数边界。
- 修复引擎在 `-WhatIf` 自检下通过日志轮转测试。
- Helper 重新编译并通过操作白名单和日志轮转自检。

## 结论

后台日志、结构化事件、龙猫云断连历史和当前状态都已具备明确上限。除手动执行生成的扫描报告和修复备份外，日常后台监控不再产生无限增长文件。
