# 多个代理客户端为什么会互相打架：两台 Windows 电脑的实测记录

## 起因

这次排查源自我之前写的[《闲聊下机场客户端和订阅阅后即焚》](https://kerrynotes.com/airport-client-choice/)。

当时我发现一个奇怪现象：机场节点测速很快，但实际使用官方客户端时，仍然可能突然打不开网页。后来我意识到，原来的测试并不够严谨，因为我的主力电脑安装过很多代理客户端。它们留下的设置、后台进程和虚拟网卡，都可能影响测试结果。

所以这一次，我把两台电脑分开记录：

| 电脑 | 实际情况 | 这台电脑适合观察什么 |
|---|---|---|
| 电脑 A：主力电脑 | 安装和使用过很多代理客户端 | 观察日常使用中会遇到哪些复杂冲突 |
| 电脑 B：实验电脑 | 以前只安装过 Clash Verge；这次才安装龙猫云和唯兔云 | 在相对简单的环境中观察问题是否还会出现 |

两台电脑不在同一个城市，但宽带都是湖南联通。这减少了“不同运营商”带来的差异，但不能把两台电脑当成完全相同的网络。不同城市仍可能使用不同的本地接入设备、DNS 和上网路线。

电脑 B 比主力电脑干净一些，但它也不是刚安装好的全新 Windows。因此，下面的内容更准确地说是一次有记录的排查测试，还不能算严格的科学对照实验。

## 先说结论

多个代理客户端之间通常没有谁天生“优先级更高”。我们感觉 Clash Verge、Clash Party 好像更强势，往往是因为：

- 它们启动或退出时，重新修改了 Windows 的代理开关；
- 它们占用了另一个客户端也想使用的端口；
- 它们启动了虚拟网卡，直接接管了更多网络流量；
- 某些应用还保留着旧代理地址，没有跟随 Windows 当前设置。

可以把它想成几个人同时使用一块路牌。最后一个改路牌的人，会决定大部分软件往哪里走；某个人退出时如果顺手把路牌撤掉，其他代理即使还在运行，浏览器也可能已经不再经过它。

所以，问题的关键不是“谁的等级更高”，而是“最后是谁改了设置，现在流量到底经过谁”。

## 我是怎样判断问题的

网页打不开时，不能马上认定是机场线路坏了，也不能只看代理客户端窗口是否还开着。我主要检查四件事：

1. 不使用代理时，普通网站能不能打开；
2. Windows 代理开关是否开启，指向哪个本机端口；
3. 这个端口实际属于哪个程序；
4. 通过这个端口访问多个网站是否成功。

这里的“端口”可以理解为本机代理的门牌号。例如 `127.0.0.1:7890` 和 `127.0.0.1:7892` 是两个不同的门牌号。Windows 只记得应该去哪个门牌号，并不知道门后面究竟是龙猫云还是 Clash Verge。

## 这几天实际观察到了哪些现象

下面先只列“做了什么”和“随后看到了什么”，不急着把责任归到某个客户端。这样可以避免把先后发生的两件事直接当成因果关系。

### 1. 龙猫运行时退出全球云，龙猫也跟着失效

当时龙猫正在正常代理。退出全球云后，龙猫的一个后台进程也退出了，随后无法通过龙猫上网。

这个现象说明，两个客户端的后台运行部分可能没有完全隔离。但是当时没有记下被关闭进程的名称、PID 和退出原因，所以目前不能直接写成“全球云向龙猫发送了停止命令”。下一次复现时需要保存退出前后的进程和端口记录。

### 2. 龙猫运行时打开 Clash Verge，网络变成直连

龙猫原本正在运行。打开 Clash Verge 后，查询出口 IP 时显示为直连。

这说明“打开另一个客户端”本身就可能改变当前网络路径，并不一定要在新客户端里手动点击“系统代理”。但这次还缺少打开前后的 Windows 代理开关和端口归属快照，所以暂时不能确定是 Clash Verge 关闭了系统代理、接管失败，还是龙猫的后台状态同时发生了变化。

### 3. Clash Verge 已经退出界面，后台核心仍可能占着 7890

扫描中出现过 Clash Verge 界面已经退出，但 `verge-mihomo` 仍在后台监听 7890 的情况。因此，“窗口关了”不等于“代理已经完全退出”。这类后台残留会影响后来打开的龙猫或其他客户端。

### 4. 只打开 Clash Party，也会启动后台核心

在龙猫已经运行时，打开 Clash Party，即使没有在 Clash Party 里开启系统代理，它仍会启动自己的 `mihomo.exe`，并监听本机端口。监控因此发现多个代理客户端同时运行，但当时 Windows 系统代理仍由龙猫接管。

这说明“没有打开系统代理”不等于这个客户端完全没有参与网络环境。它的后台核心和端口仍然存在。

### 5. 退出 Clash Party 后，龙猫核心还在，Windows 代理却被关了

退出 Clash Party 后，Windows 代理开关变成关闭。此时龙猫的后台代理程序仍在运行，本地代理端口也仍然可用，但 Chrome 开始直连，Google 无法正常访问。

与此同时，Codex 仍然可以联网，因为它还保存着旧的本地代理地址，继续直接连接龙猫的代理端口。这也是为什么会出现“浏览器不能用，但 Codex 还能用”的反常现象。

### 6. 启动部分 FlClash 同源客户端时，原来的 Clash 系统代理会被关闭

实机测试中还观察到：当 Clash 正在接管 Windows 系统代理时，启动龙猫、FlClash、肥猫云、全球云或 SpeedCat 等客户端，原来有效的 Clash 系统代理可能被关闭。普通国内网站仍可能直连打开，但需要代理的网站会失败。

这个现象说明，一些客户端在启动阶段就会清理或重设 Windows 代理，并不是点击“连接”以后才开始修改网络设置。

### 7. 龙猫和唯兔云同时存在时，龙猫后台发生过异常

电脑 B 安装龙猫和唯兔云后，记录到龙猫代理端口、内部通信端口和后台辅助服务同时异常；重启龙猫后恢复。

这个现象比单纯的“节点超时”更像本机客户端冲突，但现有记录仍不能证明是唯兔云直接关闭了龙猫。

### 8. 龙猫和 Clash Verge 可以同时运行，但不同软件可能走不同客户端

把龙猫改为 7892 后，龙猫监听 7892，Clash Verge 的后台核心仍监听 7890。Windows 系统代理指向龙猫 7892，但已经打开的 Codex 仍记着 Clash Verge 的 7890。

结果就是：浏览器走龙猫，Codex 走 Clash Verge。两个软件都能联网，但出口和规则可能完全不同。只看“能不能上网”，无法判断当前究竟是谁在代理。

把这些现象放在一起看，可以发现问题不只有一种：有时是后台进程被关闭，有时是 Windows 代理开关被改掉，有时是旧核心残留，还有时是不同应用分别走不同客户端。因此，监控程序不能只显示一个“当前主要代理”就认为一切正常。

## 两台电脑不在同一个城市，会不会影响结论

会有影响，但这不一定是坏事。两个城市相当于两个观察点，可以帮助我们判断问题是大范围发生，还是只发生在某台电脑、某个城市或某条上网路线。

虽然两边都是湖南联通，但可以把它们想成同一家快递公司的两个营业点：公司相同，不代表包裹经过的每一段道路都相同。两台电脑到龙猫入口服务器之间，仍可能走不同的本地线路或出口。

这里还有一个重要条件：龙猫目前不会根据用户所在城市分配不同的入口机器，两台电脑统一连接 Amazon 入口服务器。也就是说，这次实验基本排除了“两个城市被分配到不同入口机器”这个变量。

如果两台电脑确实在同一时间持续监控，那么断连时间不一样可以说明：

- 不像是龙猫所有用户同时掉线；
- 不像是湖南联通全省同时无法连接龙猫；
- 同一入口服务器当时不太可能处于完全宕机状态，否则两台电脑应该在接近的时间一起失败；
- 更可能是某台电脑本机、某个城市到 Amazon 入口服务器的路线，或者入口之后的某一段服务出现了局部问题。

但是，这仍然不能证明“龙猫的线路没有问题”。例如电脑 A 正常、电脑 B 断连时，仍可能是电脑 B 所在城市到 Amazon 入口服务器的路线出了问题，也可能是电脑 B 的客户端或本机代理设置出了问题。入口服务器也可能只影响部分连接，而不是对所有用户同时完全离线。

龙猫的产品设计虽然是统一使用 Amazon 入口服务器，但正式对比时仍应记录两台电脑实际连接的服务器 IP 和端口。这样才能确认实验当刻两边确实连接到了同一个入口，而不是只根据客户端显示作判断。

因此，这组现象目前最适合写成：

> 两台电脑位于不同城市，宽带同为湖南联通，并统一使用龙猫的 Amazon 入口服务器。已记录到的断连时间并不一致，因此可以初步排除入口服务器在当时发生了覆盖两个观察点的完全宕机，也不支持“湖南联通全省同时无法连接龙猫”的判断。但这不能证明龙猫线路完全正常，问题仍可能来自某台电脑、某个城市到入口服务器的网络路线，或入口之后的局部服务。

## 发现一：所有节点同时超时，不一定是客户端坏了

2026 年 7 月 14 日，电脑 B 记录到三段代理路径异常：

| 时间 | 持续时间 | 当时看到的现象 |
|---|---:|---|
| 12:11:57—12:29:34 | 约 17 分 36 秒 | 普通网络正常，本地代理端口还在，但多个代理网站打不开 |
| 12:49:36—12:49:47 | 约 10.7 秒 | 同样现象，随后自动恢复 |
| 13:28:16—13:28:27 | 约 10.7 秒 | 同样现象，随后自动恢复 |

第一段时间看起来很长，但当时没有人在电脑旁及时切换节点，所以它也包含了等待处理的时间。不能简单写成“服务器连续故障了 17 分钟”。

这三次记录只能证明代理这条路当时不通，不能证明一定是龙猫云客户端的问题。节点、线路、代理核心都有可能是原因。

## 发现二：安装龙猫云和唯兔云后，出现了本机异常

2026 年 7 月 15 日，电脑 B 又记录到两次异常：

- 08:33:03—08:34:59，约 115.2 秒；
- 08:35:54—08:36:09，约 15 秒。

这一次不仅是网站打不开。记录还显示，龙猫云负责代理的后台程序和内部通信端口同时出现异常；重启龙猫云后，它们又恢复了。

这说明问题更像是发生在电脑本机，而不只是远端节点故障。但是，当时没有记录到“哪个程序执行了什么命令”，所以仍然不能直接说是唯兔云关闭了龙猫云。

目前只能得出一个谨慎结论：

> 电脑 B 原来只安装过 Clash Verge。在加入龙猫云和唯兔云后，确实出现过龙猫云后台运行异常；但现有记录还不能确认是谁直接触发了它。

## 发现三：退出 Clash Party 后，龙猫云还在，但浏览器不走代理了

2026 年 7 月 16 日 11:21:42，电脑 A 出现了一个很有代表性的情况：

- 退出 Clash Party 后，Windows 代理开关变成关闭；
- 关闭前，Windows 代理指向 `127.0.0.1:7890`，这个端口当时属于龙猫云；
- 龙猫云的后台代理程序仍然在运行；
- Chrome 等跟随 Windows 设置的软件开始直连；
- Codex 仍能联网，因为它还记着旧的代理地址 `127.0.0.1:7890`。

这就像路边的服务站还在营业，但指向它的路牌被撤掉了。知道地址的 Codex 仍然可以过去，跟着路牌走的浏览器却不会再经过它。

从发生时间看，高度怀疑 Clash Party 在退出时关闭了 Windows 代理开关。但是当时没有开启 Windows 设置修改记录，所以现在只能说“高度怀疑”，不能说已经抓到了它修改设置的直接证据。

## 把龙猫云改成 7892 有没有用

有用，但只能解决一部分问题。

修改后，电脑 A 的情况是：

| 代理入口 | 实际程序 |
|---|---|
| `127.0.0.1:7892` | 龙猫云 |
| `127.0.0.1:7890` | Clash Verge |

两个客户端使用不同门牌号后，就不会再直接争抢同一个端口。因此，把龙猫云改为 7892，可以减少端口冲突。

但它不能解决下面这些问题：

- 另一个客户端仍可能打开或关闭 Windows 代理；
- 开启虚拟网卡的客户端仍可能接管流量；
- 某些应用仍可能记着旧的 7890；
- 两个客户端的后台服务仍可能互相影响。

实测中，浏览器跟随 Windows 设置走龙猫云 7892，而当前 Codex 仍通过旧环境变量走 Clash Verge 7890。也就是说，同一台电脑上的不同软件，可以同时走不同的代理客户端。

## 为什么这些客户端可能互相影响

龙猫云和唯兔云使用的后台程序结构，与开源客户端 FlClash 有明显相似之处。FlClash 的官方源码显示：

- 客户端退出时，会清理系统代理并关闭后台核心；
- 后台辅助服务使用固定的本机控制端口；
- 辅助服务负责启动和停止代理核心。

这可以帮助解释“为什么一个客户端退出时可能影响另一个客户端”，但源码只能作为线索。我们不能因此认定龙猫云和唯兔云保留了完全相同的代码，也不能用源码代替现场日志。

相关代码可以查看：[退出流程](https://github.com/chen08209/FlClash/blob/v0.8.83/lib/controller.dart)、[辅助服务常量](https://github.com/chen08209/FlClash/blob/v0.8.83/lib/common/constant.dart)、[后台核心管理](https://github.com/chen08209/FlClash/blob/v0.8.83/services/helper/src/service/hub.rs)和 [Windows 服务管理](https://github.com/chen08209/FlClash/blob/v0.8.83/lib/common/windows.dart)。

## 这次测试能证明什么

目前可以确认：

- 多个代理客户端可能在 Windows 代理开关、端口、虚拟网卡和后台服务等位置互相影响；
- 客户端窗口还开着，不等于它仍在接管网络；
- 后台核心还在运行，也不等于浏览器正在使用它；
- 改成 7892 可以避开端口冲突，但不能解决所有冲突；
- 浏览器和 Codex 可能同时走不同的代理。
- 两个城市使用同一 Amazon 入口服务器，但断连时间不一致，可以初步排除该入口在当时覆盖两个观察点的完全宕机。

目前还不能证明：

- Clash Verge 或 Clash Party 在 Windows 里有正式的更高优先级；
- 唯兔云确定关闭了龙猫云；
- 所有断连都来自客户端，而不是机场线路或节点；
- 龙猫线路完全没有问题；
- 龙猫云、唯兔云或 Clash Verge 中，哪一个一定更稳定；
- 改成 7892 后，多个客户端就能长期和平共存。

## 下一次怎样测试得更严谨

如果以后要真正比较客户端稳定性，应该从同一个干净的 Windows 系统备份开始：

1. 先只安装龙猫云，记录长时间运行结果；
2. 恢复到相同的系统备份，再只安装唯兔云；
3. 再次恢复系统，分别测试两个客户端同时安装、不同启动顺序和不同退出顺序；
4. 每种情况至少重复三次；
5. 两台电脑校准时间，在相同时间监控相同的网站；
6. 虽然龙猫统一使用 Amazon 入口服务器，仍要记录两边实际连接的服务器 IP 和端口，确认实验当刻确实相同；
7. 记录是谁修改了 Windows 代理开关，以及每个本机端口实际属于谁；
8. 断连时分别记录普通网络是否正常、节点服务器是否可达，以及代理访问是否成功。

这样才能把“客户端本身不稳定”“机场线路不稳定”和“多个客户端互相冲突”分开。

## 断网急救应该怎样处理

断网急救不能只判断“有没有代理客户端正在运行”，而应该回答几个更实际的问题：

- Windows 当前是否打开了代理；
- Windows 指向的端口是否真的有人监听；
- 监听程序是不是当前主要代理；
- 浏览器和 Codex 是否走了不同的代理；
- 某个客户端退出后，是否误关了其他客户端仍在使用的系统代理。

当前的处理原则是：先确认端口和程序归属，再决定是否恢复 Windows 代理。无法确认时只提示，不随便结束进程或修改网络设置。

## 总结

Clash Verge、Clash Party 看起来比龙猫云“优先级更高”，并不是因为 Windows 给它们安排了更高等级，而是因为多个客户端都能修改同一套网络设置。谁最后改了设置，谁就更像是抢到了控制权；谁退出时把公共开关关掉，也可能顺便让其他客户端失效。

把龙猫云改为 7892 值得做，因为它能避开与 7890 的直接端口冲突。但要真正避免互相打架，最稳妥的方法仍然是一次只让一个客户端接管网络。需要同时运行时，就必须继续监控 Windows 代理、实际端口归属和各个应用使用的代理地址。
